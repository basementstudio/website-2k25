## Codebase Patterns
- Package manager is `pnpm` (v9.15.6), configured via `packageManager` field in package.json
- Pre-existing typecheck errors exist related to `basehub` module exports (fragmentOn, basehub, Client, Pump) ‚Äî these are NOT caused by Sanity migration work
- Project uses React 19.0.1 (canary) and Next.js 15.6.0-canary.60 ‚Äî peer dependency warnings from Sanity packages expecting React 19.2+ are expected and non-blocking
- Existing service pattern: `src/service/<provider>/index.ts` for client setup (see `src/service/basehub/index.ts`)
- Sanity env config lives at `sanity/env.ts` ‚Äî import projectId/dataset from there (not directly from process.env)
- tsconfig path aliases: only `@/*` ‚Üí `./src/*` exists ‚Äî use relative imports for root-level `sanity/` directory
- Prefer importing `createClient` from `next-sanity` (not `@sanity/client`) for Next.js-specific features
- Import `SanityImageSource` type directly from `@sanity/image-url` (not deep path `lib/types/types`)
- Sanity schemas live at `sanity/schemas/` ‚Äî objects in `objects/`, documents in `documents/`, singletons in `singletons/`
- All Sanity schemas use `defineType` and `defineField` from `sanity` ‚Äî follow this pattern consistently
- `sanity/schemas/index.ts` exports `schemaTypes` array ‚Äî add new schemas to this array
- No project-level `typecheck` script ‚Äî use `pnpm exec tsc --noEmit` directly
- ~124 pre-existing TS errors (basehub module + page-level type issues) ‚Äî new schemas must introduce zero additional errors
- `@sanity/vision` must be installed separately for the GROQ vision tool in Sanity Studio
- `sanity.config.ts` needs `'use client'` directive since it's imported by the Studio client component
- Singleton behavior enforced via: custom desk structure (fixed documentId), template filtering, and action filtering
- Migration scripts live at `scripts/migration/` ‚Äî use `pnpm exec tsx scripts/migration/<script>.ts` to run
- Migration scripts use `dotenv/config` to load `.env.local` ‚Äî import it at the top of each entry point
- Use `@ts-expect-error` for basehub imports in migration scripts (same known canary SDK type issue)
- `@sanity/client` (not `next-sanity`) for write operations in migration scripts ‚Äî migration scripts don't need Next.js-specific features
- Image deduplication cache in `utils/images.ts` maps URL ‚Üí SanityAssetRef to avoid re-uploading same image
- Sanity slug fields must be `{ _type: 'slug', current: 'the-slug' }` ‚Äî not plain strings
- Deterministic ID pattern for migration: `{type}-{slug}` (e.g., `project-my-project`, `person-john-doe`)
- Singleton migration IDs use the schema name directly (e.g., `homepage`, `servicesPage`) ‚Äî no slug suffix
- BaseHub component fields (like `capabilities`, `ventures`) are single objects, not arrays ‚Äî query without `.items`
- When BaseHub stores plain strings but Sanity expects PT, use `plainTextToPortableText()` helper to wrap in blocks
- Sanity dataset export creates a tar.gz with a timestamped subdirectory (e.g., `production-export-2026-03-27t21-24-47-937z/data.ndjson`) ‚Äî don't assume `data.ndjson` is at root
- Pre-existing TS errors reduced from ~124 to 8 (basehub canary types may have updated) ‚Äî all in other migration scripts, not new code
- Audit results from export: 176 docs total. Gaps: 0 showcaseEntry (expected 25), 3/27 posts, 12/30 awards, 0 testimonials, 0 values, 0 homepage. 16 docs missing required fields. 0 broken refs. 12 empty PT fields.
- BaseHub client `logo` is an inline SVG string (not a URL) ‚Äî use `uploadImageToSanity(Buffer.from(svg, 'utf-8'), filename)` to upload directly instead of `downloadAndUploadImage`
- For fix scripts that need to find "missing" documents, prefer dynamic discovery (query source ‚Üí diff against Sanity) over hardcoded lists ‚Äî catches unexpected gaps
- Use `fetchExistingIds(type)` pattern (`*[_type == "X"]{ _id }` ‚Üí Set) to pre-validate references before creating documents ‚Äî avoids Sanity reference validation errors
- Site routes now live under `src/app/(site)/` ‚Äî imports must use `@/app/(site)/(pages)/` not `@/app/(pages)/`
- Studio route at `src/app/studio/` has its own minimal layout, isolated from site providers
