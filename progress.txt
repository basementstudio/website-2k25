## Codebase Patterns
- Package manager is `pnpm` (v9.15.6), configured via `packageManager` field in package.json
- Pre-existing typecheck errors exist related to `basehub` module exports (fragmentOn, basehub, Client, Pump) — these are NOT caused by Sanity migration work
- Project uses React 19.0.1 (canary) and Next.js 15.6.0-canary.60 — peer dependency warnings from Sanity packages expecting React 19.2+ are expected and non-blocking
- Existing service pattern: `src/service/<provider>/index.ts` for client setup (see `src/service/basehub/index.ts`)
- BaseHub `_title` is the internal name; `title` is the human-readable field — always prefer `title` with `_title` as fallback in migration scripts
- Migration scripts require `BASEHUB_TOKEN` AND `SANITY_API_TOKEN` env vars — cannot re-run without both
- `dotenv/config` loads `.env` by default — use `DOTENV_CONFIG_PATH=.env.local` when running migration scripts from `.env.local`
- BaseHub select fields return typed arrays — always use `Array.isArray()` check in migration scripts when writing to Sanity array fields
- Sanity MCP workspace name is `website-2k25` (not `default`) — must pass `workspaceName: 'website-2k25'` for patch/write operations
- Sanity MCP can patch/publish documents but CANNOT upload binary file assets — file uploads require SANITY_API_TOKEN and the migration scripts
- `sanity schema deploy` requires `sanity login` (interactive browser auth) — schema must be deployed via CLI after authenticating
