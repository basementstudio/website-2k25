## Codebase Patterns
- Package manager is `pnpm` (v9.15.6), configured via `packageManager` field in package.json
- Pre-existing typecheck errors exist related to `basehub` module exports (fragmentOn, basehub, Client, Pump) — these are NOT caused by Sanity migration work
- Project uses React 19.0.1 (canary) and Next.js 15.6.0-canary.60 — peer dependency warnings from Sanity packages expecting React 19.2+ are expected and non-blocking
- Existing service pattern: `src/service/<provider>/index.ts` for client setup (see `src/service/basehub/index.ts`)
- Sanity env config lives at `sanity/env.ts` — import projectId/dataset from there (not directly from process.env)
- tsconfig path aliases: only `@/*` → `./src/*` exists — use relative imports for root-level `sanity/` directory
- Prefer importing `createClient` from `next-sanity` (not `@sanity/client`) for Next.js-specific features
- Import `SanityImageSource` type directly from `@sanity/image-url` (not deep path `lib/types/types`)
- Sanity schemas live at `sanity/schemas/` — objects in `objects/`, documents in `documents/`, singletons in `singletons/`
- All Sanity schemas use `defineType` and `defineField` from `sanity` — follow this pattern consistently
- `sanity/schemas/index.ts` exports `schemaTypes` array — add new schemas to this array
- No project-level `typecheck` script — use `pnpm exec tsc --noEmit` directly
- ~124 pre-existing TS errors (basehub module + page-level type issues) — new schemas must introduce zero additional errors
- `@sanity/vision` must be installed separately for the GROQ vision tool in Sanity Studio
- `sanity.config.ts` needs `'use client'` directive since it's imported by the Studio client component
- Singleton behavior enforced via: custom desk structure (fixed documentId), template filtering, and action filtering
- Migration scripts live at `scripts/migration/` — use `pnpm exec tsx scripts/migration/<script>.ts` to run
- Migration scripts use `dotenv/config` to load `.env.local` — import it at the top of each entry point
- Use `@ts-expect-error` for basehub imports in migration scripts (same known canary SDK type issue)
- `@sanity/client` (not `next-sanity`) for write operations in migration scripts — migration scripts don't need Next.js-specific features
- Image deduplication cache in `utils/images.ts` maps URL → SanityAssetRef to avoid re-uploading same image
